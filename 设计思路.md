# AI 智能运维系统 · 设计思路

## 一、我们要解决什么问题

传统运维的痛点：

- 系统出故障时，工程师需要手动 SSH 登录、逐个检查日志、凭经验判断根因
- 排查链路长：发现报警 → 定位服务 → 分析日志 → 重启修复，每一步都靠人
- 夜间故障响应慢，对人的依赖度高

我们想做的事：**让 AI 充当一个 7×24 在线的运维工程师**，收到故障信号后自动完成「发现问题 → 分析原因 → 执行修复 → 汇报结果」的完整闭环。

---

## 二、核心思路：ReAct 模式

整个 Agent 的运作方式基于 **ReAct（Reasoning + Acting）** 模式：

```
用户提问
  ↓
[思考] 我需要先了解系统整体状态
  ↓
[行动] 调用 list_services 工具
  ↓
[观察] producer 进程不存在，PID 文件残留
  ↓
[思考] 需要看日志确认是崩溃还是被手动停止
  ↓
[行动] 调用 read_logs producer
  ↓
[观察] 日志显示正常退出，无报错，推测外部信号终止
  ↓
[行动] 调用 restart_service producer
  ↓
[观察] 重启成功，新 PID 写入
  ↓
[行动] 调用 query_database 验证数据恢复写入
  ↓
[输出] 问题报告
```

这个循环由 LLM 驱动，不需要硬编码任何 if-else 逻辑。LLM 自己决定下一步调用哪个工具、什么时候停止。

## 三、技术架构

### 整体分层

```
用户层       飞书机器人（自然语言交互）
              ↓
接入层       Webhook 服务（Flask）+ Cloudflare Tunnel（公网）
              ↓
智能层       AI Agent（DeepSeek LLM + ReAct 循环）
              ↓
工具层       6 个运维工具（进程管理、日志读取、数据库查询、指标获取）
              ↓
基础设施层   Kafka、MySQL、Redis、Prometheus（Docker 容器）
```

### 被监控的业务系统

我们专门搭了一个「模拟生产环境」供 Agent 监控：

- **Producer**：每 10 秒向 Kafka 发一条传感器数据
- **Consumer**：消费 Kafka 消息，写入 MySQL
- **Frontend**：Flask 页面展示实时数据条数（:5001）

这三个服务模拟了一条完整的数据链路，任何一个出问题都会产生可观测的故障现象。

## 四、关键技术决策

### 1. 为什么用 Tool Calling 而不是直接让 LLM 输出命令

如果让 LLM 输出 shell 命令再执行，有两个问题：
- 安全风险：LLM 可能生成危险命令
- 不可控：无法限制 LLM 的操作范围

Tool Calling 方案预先定义好 6 个工具，LLM 只能在这个范围内操作，既安全又可审计。

### 2. 为什么用 DeepSeek 而不是其他模型

DeepSeek 兼容 OpenAI 接口格式，迁移成本极低，且对中文指令和运维场景理解效果好，API 成本也更低。

### 3. 为什么用 Cloudflare Tunnel 而不是云服务器

本地 Mac 跑服务，不需要购买服务器。Cloudflare Tunnel 建立一条从本地到公网的加密通道，飞书可以直接向 `webhook.tiancaizhaozhao.dpdns.org` 发事件，且域名永久固定，不像临时隧道每次都变。

### 4. 为什么用 macOS LaunchAgent 做自启动

LaunchAgent 是 macOS 原生的服务管理机制，比手动在终端保持进程更可靠。配置后登录即自启、崩溃自动重启，飞书机器人随时可用，不需要手动操作。

### 5. 简易 RAG 而非向量数据库

我们用关键词检索实现了一个极简 RAG（检索增强生成）：把系统拓扑文档按关键词打分，把相关段落注入 system prompt。没有引入 Embedding 模型和向量库，零额外依赖，够用就好。

## 五、完整数据流

### 正常运行

```
Producer.py
  → 每10秒发 JSON {"sensor": "s1", "value": 36.5}
  → Kafka topic: sensor-data
  → Consumer.py 消费
  → 写入 MySQL sensor_data 表
  → Frontend 查询展示
```

### 故障响应

```
故障发生（进程崩溃 / 容器停止 / 数据异常）
  → 用户在飞书发消息："系统好像有问题"
  → 飞书事件推送到 Webhook（https://webhook.tiancaizhaozhao.dpdns.org/webhook）
  → Flask 服务器解析消息，异步启动 Agent
  → 在用户消息上加 🤔 reaction 表示正在处理
  → Agent ReAct 循环：调用工具 → 分析 → 继续调用 → 得出结论
  → 结果以飞书卡片形式回复（Indigo 头部 + 表格 + 分析文字）
```

## 六、故障注入设计

为了演示和测试，我们设计了 10 种故障类型，覆盖不同层次：

| 层次 | 故障类型 | Agent 表现 |
|------|----------|------------|
| 进程层 | Kill producer/consumer/frontend | 检测进程死亡 + 重启 |
| 链路层 | Kill all 三个服务 | 逐一发现并修复 |
| 基础设施层 | 停止 Kafka/MySQL 容器 | 诊断根因，提示手动恢复 |
| 数据层 | DROP TABLE | 发现写入报错，定位到表丢失 |
| 数据层 | 注入乱码消息 | 从日志噪音中识别解析错误 |
| 资源层 | CPU 压满 | 通过 Prometheus 指标发现 |
| 运维层 | 日志洪水 | 从海量错误中提取有效信息 |

## 七、主动告警：定时巡检 + 卡片交互

除了被动响应用户提问，系统还支持**主动告警**，实现真正的 7×24 监控闭环。

### 架构

```
定时调度器（每30分钟）
  → list_services + http_check 快速检查
  → 有异常：Agent 深度诊断
  → 飞书 API 发送告警卡片（红色 ⚠️）
        ↓
用户点击「立即修复」
        ↓
飞书 card_action 回调 → Webhook → Agent 修复
        ↓
飞书 API 发送结果卡片（绿色 ✅）
```

### 关键设计决策

**两段式检查**：先用 `list_services` 做快速检查（无 LLM 调用，<1秒），有问题才调 Agent 深度分析。正常情况下每次巡检零 API 费用。

**卡片回调（Card Action）**：飞书交互卡片的按钮点击会触发 POST 回调到 Webhook（`event_type: card.action.trigger`），3 秒内必须返回 toast，修复逻辑在后台异步执行。

**chat_id 持久化**：通过 `FEISHU_ALERT_CHAT_ID` 环境变量固定告警接收目标，同时自动记录最近对话的 chat_id 作为备用。

## 八、局限与可扩展方向

**当前局限：**
- 工具只覆盖了预定义的 6 个操作，Agent 无法执行工具范围外的修复
- Kafka/MySQL 容器级故障 Agent 只能诊断，不能自动 `docker compose start`（权限隔离问题）
- 知识库是静态文档，不会自动学习新的故障模式

**如果继续完善：**
- 增加 `run_docker_command` 工具，让 Agent 能管理容器
- 引入真正的向量数据库（如 Chroma），实现语义检索历史故障案例
- 接入 Prometheus 告警规则，让系统主动推送故障通知而不是等人问
- 多 Agent 协作：一个 Agent 发现问题，另一个执行修复，第三个做验收

## 九、这个项目体现了什么

用一句话概括：**LLM 不只是一个问答工具，通过 Tool Calling + ReAct，它可以成为一个真正能干活的自动化 Agent。**

传统自动化运维靠的是「规则引擎」：if CPU > 90% then restart。这种方式需要提前枚举所有可能的故障场景。

AI Agent 运维靠的是「理解 + 推理」：给它工具，它自己根据观察到的现象决定下一步怎么做，能处理没有预设规则的场景，更接近真实运维工程师的思维方式。

## 十、本地代码是如何和飞书机器人连接的

整条链路如下：

```
你在飞书发消息
      ↓
飞书服务器（open.feishu.cn）
      ↓  HTTPS POST 推送事件
Cloudflare 全球节点
      ↓  加密隧道（cloudflared 进程）
你的 Mac 本地（localhost:8080）
      ↓
Flask Webhook 服务（feishu_bot/server.py）
      ↓
AI Agent（agent/agent.py）→ DeepSeek API
      ↓
飞书 API（发送回复）
      ↓
你在飞书看到结果
```

### 关键在于 Cloudflare Tunnel

你的 Mac 没有公网 IP，飞书服务器无法直接访问你的电脑。Cloudflare Tunnel 解决了这个问题：

```
Mac 本地                          Cloudflare 云端
┌─────────────────┐               ┌──────────────────────────────┐
│ cloudflared 进程 │←─── 长连接 ──→│ webhook.tiancaizhaozhao      │
│ （主动发起连接） │               │ .dpdns.org                   │
└─────────────────┘               └──────────────────────────────┘
        ↓ 转发到本地                        ↑
  localhost:8080               飞书推送事件到这个公网域名
```

`cloudflared` 进程在你 Mac 上**主动**和 Cloudflare 建立一条加密的持久连接。有请求进来时，Cloudflare 通过这条连接把数据转发到你本地的 8080 端口。不需要开放任何防火墙端口，也不需要公网 IP。

### 飞书那边的配置

飞书开发者后台「事件订阅」里填了这个地址：

```
https://webhook.tiancaizhaozhao.dpdns.org/webhook
```

每次有人给机器人发消息，飞书都会往这个地址 POST 一个 JSON 事件，内容包含消息文本、chat_id、message_id 等。

### 本地代码怎么处理

`feishu_bot/server.py` 里的 Flask 收到这个 POST 之后：

```python
@app.route('/webhook', methods=['POST'])
def webhook():
    data = request.json          # 解析飞书发来的 JSON
    message_id = ...             # 提取消息 ID
    text = ...                   # 提取用户说的话
    chat_id = ...                # 提取聊天室 ID

    # 异步处理，先回 200 给飞书（飞书要求 3 秒内响应）
    threading.Thread(target=_handle, args=(text, chat_id, message_id)).start()
    return jsonify({'code': 0})
```

然后 `_handle` 在后台调用 Agent，Agent 分析完后再调飞书的发消息 API 把结果发回去。

### 为什么不用每次手动开隧道

因为 `cloudflared tunnel run ai-ops-bot` 被配置成了 macOS LaunchAgent：

```
~/Library/LaunchAgents/com.aiops.cloudflared.plist
```

登录 Mac 后自动启动，崩溃后自动重启，域名永远是同一个，所以飞书后台只需要配一次 URL 就够了。
